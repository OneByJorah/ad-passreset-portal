## What's already implemented (verified, do NOT rebuild)

The `SetPassword` fallback is live in `src/PassReset.PasswordProvider/PasswordChangeProvider.cs`
(lines ~566–604). Confirmed behavior:

- On a `COMException` from the user-initiated `ChangePassword`, the provider **first classifies
  well-known HResults** (BUG-002 / STAB-004) — notably the minimum-password-age rejection — so
  those are returned as mapped errors and **never** routed through `SetPassword` (which would
  bypass AD password history). *(line ~568–569)*
- `SetPassword` is attempted **only** when `!UseAutomaticContext && AllowSetPasswordFallback`
  (explicit-credentials mode with the flag on). Otherwise the failure surfaces as the mapped
  error. *(line ~591)*
- The fallback logs that it bypasses AD password history. *(line ~604)*

So the gap is **not** behavior — it is the absence of (a) a dedicated regression test pinning
this, and (b) the operator-facing delegation + trade-off documentation.

### Task 2: Document the SetPassword delegation + trade-off

**Files:**
- Modify: `docs/AD-ServiceAccount-Setup.md`

- [ ] **Step 1: Read the existing doc** to match its structure/voice.

- [ ] **Step 2: Add a "SetPassword fallback (`AllowSetPasswordFallback`)" subsection** covering:
  - **When it applies:** only with explicit LDAP credentials (`UseAutomaticContext = false`) and
    `AllowSetPasswordFallback = true`; never under automatic (domain-joined) context.
  - **AD right required:** the service account needs the **Reset Password** extended right on the
    target OU (distinct from the user-self **Change Password** right the normal flow uses).
  - **The trade-off (call out clearly):** `SetPassword` is an administrative reset that
    **bypasses AD password-history enforcement** — a user could set a recently-used password. The
    portal still classifies and blocks the minimum-password-age rejection *before* falling back
    (so min-age is never bypassed), but history is. Default is **off** for this reason.
  - **Recommendation:** leave it disabled unless a specific COMException-on-change scenario in the
    operator's environment requires it.
  - Cross-link from `docs/appsettings-Production.md`'s `AllowSetPasswordFallback` key if one isn't
    already present.
